DNS in NMState Context

**Two Levels of DNS Configuration**

**Critical distinction:**


**1. Host-level DNS (NMState manages this)**

/etc/resolv.conf on RHCOS nodes
  - Used by node-level processes (kubelet, CRI-O, system services)
  - Resolved via systemd-resolved or NetworkManager


**2. Cluster-level DNS (CoreDNS, NMState does NOT manage this)**

DNS for Pods and Services
  - Managed by cluster-dns-operator
  - CoreDNS running in openshift-dns namespace
  - Separate from host DNS configuration


NMState configures node-level DNS only.

DNS in IPI

**Default IPI DNS Behavior**

IPI on AWS example
Cloud provider configures DNS automatically via:
- VPC DHCP Options Set
- Instance metadata service

Check node DNS configuration
```bash
oc debug node/worker-0
chroot /host
cat /etc/resolv.conf
```

Typical IPI output (AWS):
Generated by NetworkManager
search us-east-1.compute.internal
nameserver 10.0.0.2  # ← VPC DNS resolver (Route 53 Resolver)
options ndots:5

**How it works**

1. AWS assigns DNS server via DHCP (VPC DNS resolver at VPC CIDR +2)

2. NetworkManager on RHCOS picks up DHCP DNS configuration

3. Node resolves DNS via AWS Route 53 Resolver

4. Route 53 Resolver can resolve:

Public DNS (forwarded to internet DNS)
  - Private Route 53 hosted zones
  - VPC-internal DNS names


**Similar for other clouds**
- Azure: Azure-provided DNS (168.63.129.16) via DHCP
- GCP: Google metadata server (169.254.169.254) via DHCP
- vSphere: DNS servers configured in vCenter DHCP

When to Use NMState for DNS in IPI


**Use Case 1: Override Cloud DNS for Corporate DNS**

Corporate policy: All nodes must use corporate DNS servers
Override cloud-provided DNS

```yaml
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
name: corporate-dns
spec:
nodeSelector:
node-role.kubernetes.io/worker: ""
desiredState:
dns-resolver:
config:
search:
- corp.example.com
```

- prod.corp.example.com
  server:
- 10.100.0.53      # Corporate DNS server 1
- 10.100.0.54      # Corporate DNS server 2

**Result on nodes**
```bash
cat /etc/resolv.conf
# search corp.example.com prod.corp.example.com
# nameserver 10.100.0.53
# nameserver 10.100.0.54
```

**Why you might do this**
- ✅ Corporate DNS provides access to internal services
- ✅ DNS-based content filtering/security policies
- ✅ Centralized DNS logging/auditing
- ✅ Custom internal zones (e.g., .internal, .corp)

**Considerations**
- ⚠️  Corporate DNS must resolve cloud-internal names (VPC DNS zones)
- ⚠️  Or use conditional forwarding in corporate DNS to cloud DNS

---

**Use Case 2: Hybrid Cloud - Split-Horizon DNS**

Nodes need to resolve both:
- Cloud-internal DNS (AWS Route 53 private zones)
- On-premises DNS (via Direct Connect/VPN)

Use routing rules for DNS resolution
```yaml
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
name: split-horizon-dns
spec:
nodeSelector:
node-role.kubernetes.io/worker: ""
desiredState:
dns-resolver:
config:
search:
- us-east-1.compute.internal  # AWS internal
```

- corp.example.com            # Corporate
  server:
- 10.0.0.2      # AWS VPC DNS (primary)
- 10.200.0.53   # Corporate DNS (secondary, via Direct Connect)

**DNS resolution order**

1. Try AWS VPC DNS first (can resolve AWS internal + public DNS)

2. Fall back to corporate DNS if AWS DNS fails

3. Corporate DNS has conditional forwarders:

*.compute.internal → forward to 10.0.0.2 (AWS VPC DNS)
  - *.corp.example.com → resolve locally

Multi-tenant environment with team-specific DNS zones

master-1.cluster.example.com    A     192.168.1.11

_etcd-server-ssl._tcp.cluster.example.com  SRV  0 10 2380 etcd-0.cluster.example.com

`worker.ign` snippet (traditional UPI alternative to `agent-config.yaml`):

```yaml
variant: fcos
version: 1.4.0
storage:
  files:
    - path: /etc/NetworkManager/conf.d/99-custom-dns.conf
      mode: 0644
      contents:
        inline: |
          [main]
          dns=none
    - path: /etc/resolv.conf
      mode: 0644
      overwrite: true
      contents:
        inline: |
          search cluster.example.com prod.example.com
          nameserver 192.168.1.53
          nameserver 192.168.1.54
          options ndots:5
```

**Limitations**
- ❌ Static file, not declarative
- ❌ No drift detection
- ❌ Harder to update (need to recreate nodes)
- ❌ NetworkManager might overwrite on reboot

Better approach: use `kubernetes-nmstate` operator post-install.

UPI Post-Install: Declarative DNS Management

After cluster is running, use NMState operator for DNS management.

```yaml
apiVersion: nmstate.io/v1

metadata:
  name: production-dns
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ""
  desiredState:
    dns-resolver:
      config:
        search:
          - cluster.example.com
          - prod.example.com
        server:
          - 192.168.1.53
          - 192.168.1.54
```

          - 192.168.1.53

**Advantages**

Common DNS Issues

Issue 1: Node cannot resolve `api.cluster.example.com`

Symptom:
```bash
oc debug node/worker-0
chroot /host
nslookup api.cluster.example.com
# ** server can't find api.cluster.example.com: NXDOMAIN
```

Diagnosis:
```bash
cat /etc/resolv.conf
# nameserver 192.168.1.53
dig @192.168.1.53 api.cluster.example.com
# Check if DNS server has the record
```

Fix:
- Ensure DNS server has required cluster records.
- Or update NMState policy with the correct DNS server.

---
Issue 2: DNS policy not applied

Check enactment status:
```bash
oc get nnce -o wide
# STATUS: NodeNetworkConfigurationPolicyFailing
```

Get details:
```bash
oc describe nnce worker-0-production-dns
# Error: DNS server not reachable
```

Fix:
```bash
ping 192.168.1.53
```

---
Issue 3: DNS search domain list too long

Symptom: DNS queries are slow.  
Cause: too many search domains.

Example (too many domains):
```yaml
dns-resolver:
  config:
    search:
      - cluster.example.com
      - team-a.example.com
      - team-b.example.com
      - shared.example.com
      - prod.example.com
      - corp.example.com
```

Impact:
- Short name `database` may try multiple domains before failing.

Fix (keep 3-4 relevant domains):
```yaml
dns-resolver:
  config:
    search:
      - cluster.example.com
      - prod.example.com
    server:
      - 192.168.1.53
```

DNS vs CoreDNS (Cluster DNS)

Critical distinction:

```text
Node-level DNS (NMState manages):
- Uses node /etc/resolv.conf
- Used by kubelet, CRI-O, and node services
- Resolves API endpoints, registries, and external names

Pod-level DNS (CoreDNS manages):
- Uses pod /etc/resolv.conf (dns-default service)
- Used by application pods
- Resolves service-name.namespace.svc.cluster.local and forwarded external names
```

---
Example: DNS Flow for Different Queries

1. Pod queries Kubernetes service:
```bash
kubectl exec -it mypod -- nslookup kubernetes.default
```
Flow: Pod -> CoreDNS -> answer from cluster zone.

2. Pod queries external name:
```bash
kubectl exec -it mypod -- nslookup google.com
```
Flow: Pod -> CoreDNS -> upstream node DNS -> external resolution.

3. Node process queries API:
```bash
nslookup api.cluster.example.com
```
Flow: Node process -> node DNS from `/etc/resolv.conf`.

Summary: DNS in IPI vs UPI

```text
Aspect                     | IPI                             | UPI Bare Metal                    | UPI Cloud
---------------------------|----------------------------------|-----------------------------------|-------------------------
Node DNS source            | Cloud DHCP (automatic)          | Manual config (NMState)           | Cloud DHCP or manual
DNS servers                | Cloud-provided                  | Your DNS servers                  | Cloud or your DNS
Cluster DNS records        | Installer-created               | You create before install         | You create before install
NMState role for DNS       | Optional override               | Essential for node DNS            | Optional override
Installation-time config   | cloud-init                      | agent-config.yaml (NMState)       | cloud-init or manual
Post-install management    | NMState operator (optional)     | NMState operator (recommended)    | NMState operator (optional)
Cluster DNS (CoreDNS)      | Same                            | Same                              | Same
```

---
Key Takeaways

Node DNS (NMState):
- IPI: Cloud provides DNS automatically; NMState is usually for overrides.
- UPI: NMState is critical for configuring node resolvers.
- Scope: node-level processes only (`kubelet`, `CRI-O`, system services).

Cluster DNS (CoreDNS):
- Both IPI and UPI: managed by `cluster-dns-operator`.
- NMState does not manage cluster/pod DNS.

Best practices:
1. UPI: configure DNS via NMState (`agent-config.yaml` or operator policy).
2. IPI: use NMState only when cloud DNS override is required.
3. Keep search domains small (3-4 preferred).
4. Include IPv4 and IPv6 DNS servers for dual-stack clusters.
5. Configure at least two DNS servers for HA.
6. In air-gapped environments, pre-create all required internal DNS records.
7. Validate DNS from nodes before workload rollout.
